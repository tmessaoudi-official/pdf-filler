# Dependency Upgrade to Latest Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Upgrade all 5 outdated packages to their latest major versions while keeping the app fully working.

**Architecture:** One upgrade per task, safest-first order. Each task installs, fixes breaking changes, verifies type-check + tests + `vite build`, then commits. If a step fails the build or tests the task is not done â€” diagnose, fix, re-verify. No bundling unrelated changes into the same commit.

**Tech Stack:** TypeScript 6, ESLint 10, Vite 8, vite-plugin-pwa 1.x, pdfjs-dist 6, pdf-lib 1.17.1 (unchanged), Vitest 4, jsdom 29.


  Key things to check:
  - Any `tsconfig.json` options that were removed or renamed
  - Any strict-mode changes that affect inference
  - Whether `allowImportingTsExtensions` + `moduleResolution: bundler` combo is still valid

- [ ] **Step 2: Bump TypeScript in package.json**


## Task 4: pdfjs-dist 3 â†’ 6

**Files:**
- Modify: `package.json`
- Modify: `js/pdfRenderer.ts` (worker path + import pattern changed)
- Modify: `js/pdfEditorApp.ts` (type fixes if API changed)
- Modify: `js/textSearchHandler.ts` (type fixes if API changed)

**Context:** This is the highest-risk upgrade. pdfjs-dist v3â†’v4 switched to full ES modules, changing the worker path from `build/pdf.worker.min.js` to `build/pdf.worker.min.mjs`. v4â†’v5 and v5â†’v6 may have further API and type changes. Our codebase uses:
- `pdfjsLib.GlobalWorkerOptions.workerSrc` â€” worker setup (needs path change)
- `pdfjsLib.getDocument(bytes).promise` â€” loading PDFs (API should be stable)
- `PDFDocumentProxy`, `PDFPageProxy` â€” types imported from `pdfjs-dist`
- Canvas rendering via `page.render({ canvasContext, viewport })`
- Text content via `page.getTextContent()`

The `getDocument` and rendering APIs are stable across v3â†’v6. The worker path and import approach are the main breaking change.

- [ ] **Step 1: Read the pdfjs-dist v4, v5, and v6 changelogs**

  ```bash
  # Check releases at: https://github.com/mozilla/pdf.js/releases
  # Filter for tags v4.x.x, v5.x.x, v6.x.x
  # Also check the wiki: https://github.com/mozilla/pdf.js/wiki/Setup-PDF.js-in-a-website
  ```

  Key things to check:
  - Worker import/path changes between v3 and v4
  - Any removed or renamed exports in the main module
  - TypeScript type changes (`PDFDocumentProxy`, `PDFPageProxy`, `TextContent`, etc.)
  - Whether `getDocument()` still accepts `Uint8Array` directly

- [ ] **Step 2: Bump pdfjs-dist in package.json**

  Edit `package.json`:
  ```json
  "pdfjs-dist": "^6.0.0",
  ```

- [ ] **Step 3: Install**

  ```bash
  npm install
  ```

  Expected: installs `pdfjs-dist@6.x.x`.

- [ ] **Step 4: Fix the worker path in pdfRenderer.ts**

  Current code at `js/pdfRenderer.ts:6-9`:
  ```ts
  pdfjsLib.GlobalWorkerOptions.workerSrc = new URL(
    'pdfjs-dist/build/pdf.worker.min.js',
    import.meta.url
  ).href;
  ```

  In pdfjs-dist v4+, the worker file is `.mjs` not `.js`. Also, for Vite, the recommended approach changed. Apply **one** of these fixes depending on what pdfjs-dist v6 provides:

  **Option A â€” direct URL (works if `build/pdf.worker.min.mjs` exists in v6):**
  ```ts
  pdfjsLib.GlobalWorkerOptions.workerSrc = new URL(
    'pdfjs-dist/build/pdf.worker.min.mjs',
    import.meta.url
  ).href;
  ```

  **Option B â€” Vite `?url` import (cleaner, Vite-idiomatic):**
  ```ts
  import workerSrc from 'pdfjs-dist/build/pdf.worker.min.mjs?url';
  // At the top of the file, before the class definition:
  pdfjsLib.GlobalWorkerOptions.workerSrc = workerSrc;
  ```

  To pick the right option: check what files exist in `node_modules/pdfjs-dist/build/` after install:
  ```bash
  ls node_modules/pdfjs-dist/build/ | grep worker
  ```
  Use the filename that actually exists. If both `.js` and `.mjs` exist, prefer `.mjs`.

- [ ] **Step 5: Run type-check**

  ```bash

  ```

  Fix any type errors. Common issues after pdfjs-dist major version bumps:
  - `TextItem` type may have changed (check `textSearchHandler.ts` which uses `getTextContent()`)
  - `RenderTask` type may have changed
  - `PageViewport` may have different properties
  - `getDocument()` parameter type â€” if it no longer accepts raw `Uint8Array`, wrap in `{ data: bytes }`

  For each error, check the actual type by running:
  ```bash
  cat node_modules/pdfjs-dist/types/src/pdf.d.ts | grep -A5 "TypeName"
  ```

- [ ] **Step 6: Run tests**

  ```bash

  ```

  Expected: 57 passed. Most tests mock the canvas and don't use pdfjs-dist directly, so failures here would indicate import resolution issues rather than API changes.

- [ ] **Step 7: Smoke test in the browser**


  1. Upload any PDF
  2. Verify it renders on the canvas (no blank page, no console errors)
  3. Check browser console for worker errors (e.g., "Failed to load worker script")
  4. Navigate pages (prev/next)

  If the PDF renders blank or there are worker 404 errors:
  - Check the network tab for what URL the worker is being fetched from
  - Verify `ls dist/` after `npm run build` contains the worker file
  - Try Option B (Vite `?url` import) if Option A didn't work

- [ ] **Step 8: Run build**
